# 마크업 표기 룰 + 4주치 샘플 레이아웃 학습 정책

> **위치 (정본)**: `Congregation/.claude/shared/markup-and-layout-policy.md`
> **작성 (2026-04-25)**: chair 작업 중 본체가 휴리스틱을 떠맡는 문제 발견 → 기획자(에이전트) 가 직접 분석·마크업 명시·1:1 변환하도록 영구 정책화.

## 적용 대상 에이전트

본 정책은 다음 회중 에이전트가 **자기 정의 본문에서 직접 참조 + 의무 Read** 한다:

- `chair-script-builder` — 사회자 전체 대본 (4주치 샘플: 231213/250718/251031/260311)
- `prayer-composer` — 시작·마침 기도 (chair 산출물 [블록 0a]·[블록 9a] 위치 삽입)
- (확장 가능) 다른 script 에이전트 — 자기 파트 실전 샘플이 존재하는 경우

새 에이전트 추가 시: 자기 정의 본문에 다음 문장 박기 — *"본 정책 파일 (`Congregation/.claude/shared/markup-and-layout-policy.md`) 을 먼저 Read 한 뒤 작업한다."*

## 핵심 원칙

1. **본체(Claude 메인) 또는 빌더(build_*.py)가 휴리스틱·자동 추측으로 강조·하이라이트·들여쓰기를 결정하지 않는다.**
2. **기획 에이전트가 직접 4주치 샘플 분석 → meta.yaml `layout_patterns` 에 명시 → ④ Script 본문에 마크업 표기 → 빌더는 1:1 변환만.**
3. 휴리스틱이 들어가면 정확도가 들쑥날쑥, 정형구 바뀌면 깨짐, 새 패턴 추가 시 빌더 수정 필요. 기획자가 마크업으로 명시해야 정확.

## ① 단계 — 실전 샘플 의무 Read

각 파트 기획 에이전트는 ① Planner 단계 진입 즉시 **자기 파트의 4주치 (또는 그 이상) 실전 샘플 docx** 분석 (paragraph 별 폰트·크기·색상·하이라이트·정렬·들여쓰기 추출).

### chair (사회자 전체 대본) — 샘플 4개

```
~/Dropbox/02.WatchTower/01.▣ 수원 연무 회중/01.주중집회/00.생봉 사회/
├── 20231213/생봉 집회 사회 20231213_김원준.docx
├── 20250718/생봉 집회 사회 20250718_김원준.docx
├── 20251131/생봉 집회 사회 20251031_김원준.docx
└── 20260312/생봉 집회 사회 20260311_김원준.docx
```

분석 도구: `python-docx` (Windows python). JSON 캐시 (`/tmp/chair_sample_*.json`) 가 이미 있으면 재사용 — 새로 추출하지 않음.

### 다른 파트

각 에이전트는 자기 파트 산출물 폴더 (예: `01.성경에 담긴 보물/01.10분 연설/`) 의 과거 docx 를 동일 방식으로 분석.

## meta.yaml 의 `layout_patterns` 키 (필수)

분석 결과를 다음 형태로 meta.yaml 에 명시:

```yaml
layout_patterns:
  body_size_pt: 13
  direction_size_pt: 8                # < 무대 지시 > 8pt 회색
  inline_quote_size_pt: 12            # 성구·요점 verbatim 인용 12pt (선택)
  time_marker_color: "FF0000"
  time_marker_size_pt: 13
  time_markers_standard: [...]
  time_markers_sub: [...]              # 보조 마커 (들여쓰기 적용)
  indent_emu_standard: 508000          # 인용·실연·성구 verbatim 들여쓰기 단일 표준
  yellow_highlight_targets: [...]      # 노랑 적용 대상 카테고리 목록
  red_bold_keywords: [...]             # 빨강 볼드 키워드 목록
  block_headers: false                 # [블록 N] 같은 표지 docx 출력 여부
  fixed_section_transitions: [...]     # 고정 전환 정형구 목록
  prayer_full_text: true               # 시작·마침 기도 전문 작성 여부
  next_week_assignment_format: "..."   # 임명 호명 형식
  closing_signature: "..."             # 마무리 시그너처
  # ── 줄간격·여백 표준 (4주치 샘플 평균) ──
  paragraph_space_before_pt: 0         # 단락 위 여백
  paragraph_space_after_pt: 0          # 단락 아래 여백 (4주치 모두 0~1pt 빡빡)
  line_spacing: 1.15                   # 줄 간격 (4주치 모두 빡빡)
  time_marker_space_after_pt: 2        # 시간 마커는 그 다음 단락과 약간 간격
  box_shading_color: "F2F2F2"          # [box] 회색 바탕
  box_border_color: "808080"           # [box] 테두리 회색
```

각 파트별로 어떤 키가 필수인지는 해당 에이전트 정의 본문에서 추가 명시.

## 줄간격·여백 표준 (4주치 샘플 분석 — 영구)

⚠ **빈 단락 / 큰 space_after / 넉넉한 line_spacing 일체 금지**. 원준님 4주치 샘플은 **빡빡한 레이아웃** 으로 한 페이지에 많은 본문이 들어가는 형태. 줄을 띄엄띄엄 출력하면 사회자가 페이지를 넘기면서 진행 흐름 끊김.

빌더 표준:
- `space_before = Pt(0)` (모든 단락)
- `space_after = Pt(0)` (일반 단락)
- `space_after = Pt(2)` (시간 마커, 그 다음 단락 시작 직전 약간 간격)
- `line_spacing = 1.15` (한 줄 안 행 높이)
- 빈 줄 (마크다운의 `\n\n`) 도 docx 에서는 **새 단락 1개만** 추가, space_after Pt(0). 즉 마크다운 빈 줄이 docx 큰 공간으로 변환되지 않음.

## ④ Script 단계 — 마크업 표기 룰 (전 에이전트 공통)

본 에이전트가 ④ 단계에서 작성하는 마크다운 본문에 다음 마크업을 직접 명시. 빌더(`build_*.py`)는 휴리스틱·자동 추측 일체 없이 이 마크업만 그대로 docx 로 변환.

| 표기 | 의미 | docx 변환 |
|---|---|---|
| `==text==` | 노란 하이라이트 + 볼드 | YELLOW highlight + bold |
| `**text**` | 굵게 (하이라이트 없음) | bold |
| `*text*` | 이탤릭 | italic |
| `[red]text[/red]` | 빨강 볼드 (예: "성경 낭독 시간", "(필수)") | color #FF0000 + bold |
| `[blue]text[/blue]` | 파랑 볼드 (예: 성구 강조) | color #2F5496 + bold |
| `> text` | 들여쓰기 (508000 EMU = 표준) — 인용·실연·성구 verbatim | left_indent 508000 |
| `< text >` | 무대 지시 — 8pt 회색 이탤릭 | size 8pt + #606060 + italic |
| `By H:MM pm` (단독 라인) | 시간 마커 | RIGHT 정렬 + #FF0000 + bold + 13pt. 보조 마커 (예: 8:04/8:05/8:09/8:10) 는 자동 들여쓰기 |
| `[box]…[/box]` (블록) | **글 상자 + 회색 바탕 (#F2F2F2) + 검은 테두리 + 들여쓰기** — 성경 낭독 verbatim·중요 인용·시인성 강조 영역 | paragraph shading + border + 13pt 그대로 |
| `[블록 N] …` 또는 `### [블록 …]` | **마크다운 작성용 골조 — docx 출력 단계에서 빌더가 skip** | 출력 X |
| `# / ## / ### …` (블록 헤더 외 메타 헤더) | 메타 노트 — 빌더 skip | 출력 X |

### 노랑 하이라이트 사용 위치 (chair 기준 예시)

④ Script 작성 시 다음 위치에 `==…==` 마커를 명시:
- 학생·연사 이름 + 호칭: `==김영준 형제==`
- 따옴표 안 제목: `=="200년 앞서 기록된 예언"==`
- 출판물 표제: `==「사람들을 사랑하십시오 — 겸손」 요점 3==`
- 조언과 번호·제목: `==4. 대화 시작하기==`
- (N분이내) 시간 표기: `==(3분이내)==`
- 환영사 핵심 문장 한 단락 전체에 노랑 — `==…==` 로 단락 처음부터 끝까지 감싸기

다른 파트는 자기 샘플 분석 결과의 `yellow_highlight_targets` 따라.

### ⚠ 마커 인접 금지 (필수)

**같은 줄에서 `==…==` 마커가 둘 이상 연속될 때, 마커 사이에는 반드시 일반 텍스트(공백·조사·문장 토큰) 가 들어가야** 합니다. 마커끼리 직접 붙으면 정규식 짝짓기 실패 → docx 에 `==` 그대로 노출 (빌더 버그).

**올바른 예** (조사·공백 사이에 일반 텍스트):
```
=="영적 보물 찾기"==를 ==(미정) 형제==께서 준비하셨습니다. ==(10분)==
```

**잘못된 예** (`==`가 인접):
```
=="영적 보물 찾기"를 ==(미정) 형제==께서 준비하셨습니다. ==(10분)==
```
(첫 마커가 `=="영적 보물 찾기"를 ==` 까지 잡혀 두 번째 마커 짝짓기가 깨짐.)

같은 룰이 `**굵게**` · `*이탤릭*` · `[red]…[/red]` · `[blue]…[/blue]` 모든 인라인 마커에 적용. 마커 종료 후 다음 마커 시작 전에 **최소 1자 이상의 일반 텍스트** (공백 포함) 의무.

## 빌더 책임 (`build_*.py`)

- 마크다운 ↔ docx **1:1 변환**만. 정형구·이름·따옴표 등 휴리스틱 자동 매칭 일체 금지.
- 본 에이전트가 마크업으로 명시하지 않은 텍스트는 일반 본문(13pt 맑은 고딕)으로 그대로 출력.
- `[블록 N]` 헤더 / 메타 노트 헤더는 정규식으로 skip.
- 페이지 여백·기본 폰트·시간 마커 색상·들여쓰기 EMU 값은 layout_patterns 그대로.
- **줄간격·단락 여백** 표준 적용 (위 §줄간격·여백 표준).
- **`[box]…[/box]` 처리**: 회색 바탕 `#F2F2F2` + 회색 테두리 `#808080` (sz=4) + 들여쓰기 `INDENT_STD` + 본문 13pt 그대로. 박스 안 텍스트는 기존 마크업 (==노랑== 등) 그대로 처리하되, 성경 낭독은 보통 마크업 없이 일반 13pt 본문.

## 보조 자료 위임 + 부족 항목 사용자 요청 (필수)

각 기획 에이전트는 ① 단계에서 **wol-researcher 적극 활용** 의무:

- 학생 과제 학습 요점 본문 verbatim (예: 「읽가」 N과 본문)
- 영보 질문 부연 설명·관련 출판물 단락
- 생활 파트 토의 질문·동영상 핵심 내용
- CBS 단락 범위·핵심 질문
- 다음 주 주제·낭독 범위

wol-researcher 가 가져올 수 없는 항목 (예: 제휴 사이트가 막힌 일부 출판물, 비공개 자료, 사용자 수동 제공 슬롯) 은 **마지막에 "사용자 입력 필요" 섹션** 자동 생성:

### ④ Script 본문 끝에 자동 첨부

`research-{part}/{YYMMDD}_*.md` 본문 마지막에 다음 섹션 추가 (마크다운):

```markdown
## ⚠ 원준님 입력 필요 (빌더는 이 섹션을 docx 끝에 자동 출력)

다음 항목은 wol·공식 출판물에서 직접 확인하지 못해 placeholder 상태입니다.
사용 전 채워주시면 ver+1 빌드로 완성됩니다.

- [ ] 「읽가」 5과 본문 verbatim (성경 낭독 학습 요점) — `[wol 확인 필요]`
- [ ] 「읽가」 16과 본문 verbatim — `[wol 확인 필요]`
- [ ] 시작 노래 12번 한국어 제목 — `(미정)`
- [ ] ...
```

빌더는 이 섹션을 docx 끝에 회색 배경 박스 + 빨강 헤더로 출력해서 원준님이 즉시 식별 가능하도록.
