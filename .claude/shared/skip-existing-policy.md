# 산출물 존재 시 skip 정책

> 버전: 1.0 / 도입: 2026-04-25 / 적용 대상: 모든 planner 에이전트 + 모든 스킬

---

## 1. 핵심 규칙

**모든 planner·스킬은 작업을 시작하기 전에 출력 폴더의 산출물을 확인해야 한다.**

| 상황 | 행동 |
|---|---|
| 산출물 없음 | 정상 진행 (신규 생성) |
| 산출물 있음 + 사용자 지시 모호 | **단정형 확인 1회** → 답 없거나 No → **skip** |
| 산출물 있음 + 사용자가 "재생성/다시/업그레이드/버전 올려" 명시 | **새 버전 번호 부여 후 신규 생성** (기존 파일 보존) |

---

## 2. 단정형 확인 메시지 형식

기존 산출물이 있으면 **메뉴 나열식이 아니라 단정형으로** 한 번 묻고, 답이 없거나 No면 skip 한다:

```
{파트} 산출물이 이미 있습니다 ({파일 경로}, mtime {YYYY-MM-DD HH:MM}).
새로 만들지 않고 그대로 두라는 말씀이시죠?
```

원준님이 답을 안 하시면 skip 이 기본값. "응", "응 skip", 답 없음 → skip.
"아니, 새로 만들어", "다시 만들어", "버전 업그레이드", "ver 올려" → 신규 생성 (버전 번호 부여).

---

## 3. 일괄 스킬 (`/weekly`, `/midweek-now/next1/next2/next3`)에서의 작동 — 묶음 확인

일괄 스킬은 한 번에 여러 파트(주차 × 파트)를 만든다. 각 파트마다 단정형 확인을 따로 묻지 않고, **시작 시점에 한 번에 표로 보고하고 사용자에게 한 번만 yes/no 받는다.**

### 시작 시 묶음 표 보고 형식

```
/midweek-next1 산출물 현황:
| 파트 | 주차 | 상태 |
|---|---|---|
| 10분 연설 | 260430 | ✅ 있음 (260430 11:42) |
| 영적 보물 | 260430 | ✅ 있음 |
| 학생과제 #1 | 260430 | ❌ 없음 |
| 학생과제 #2 | 260430 | ❌ 없음 |
| ... | | |

기존 N개는 그대로 두고, 빠진 M개만 새로 만들면 되시죠?
```

### 사용자 응답에 따른 분기

| 응답 | 분기 |
|---|---|
| 답 없음 / "응" / "그래" | 빠진 것만 신규 생성, 기존 보존 |
| "아니, 전부 새로 만들어" / "버전 업그레이드" | 모두 버전 +1 부여 후 신규 생성 |
| "특정 파트만 다시" | 명시한 파트만 버전 +1 신규 생성 |

### 자리 비움 시 안전성

원준님이 자리 비우셨을 때도 답 없음 → skip 으로 자동 작동. 검수·발송된 docx 가 의도치 않게 덮어쓰이지 않는다.

### 예시

`/weekly` 실행 시 이미 2주치 docx 있음 + 1주치 빠진 경우:
- 묶음 표 보고 → 사용자 한 번 yes
- next1 (이번 주): docx 있음 → skip
- next2 (다음 주): docx 있음 → skip
- next3 (다다음 주): docx 없음 → 신규 생성

## 4. 버전 번호 부여 규칙

기존 파일이 있고 사용자가 재생성 명시 시:

- 기존 파일명: `여호와를 하느님으로 둔 우리는 행복합니다_ver3_260430.docx`
- 신규 파일명: `여호와를 하느님으로 둔 우리는 행복합니다_ver4_260430.docx` (ver 한 단계 증가)

기존 파일에 버전 번호 없으면 `_ver2` 부여:

- 기존: `Talk5_종교단체에속해야합니까_260430.docx`
- 신규: `Talk5_종교단체에속해야합니까_ver2_260430.docx`

기존 파일은 **보존**(삭제·이동 금지). 사용자가 명시적으로 정리 요청 시에만 `_v_old/` 등으로 이동.

## 5. 확인 절차 (planner 진입 단계)

각 planner 는 작업 시작 전 다음 순서를 따른다:

1. **산출물 폴더 식별** — 자기 스킬의 정의된 출력 경로 (`research-plan/{파트}/{YYMMDD}*/` 또는 `주중집회/{경로}/`).
2. **존재 확인** — Glob/Bash 로 해당 폴더의 docx·script.md·outline.md 등 핵심 산출물 존재 여부 검사.
3. **분기**
   - 없음 → 정상 진행 (이 정책 끝)
   - 있음 + 사용자 무명시 → 단정형 확인 메시지 1회 발송 → 응답 대기
     - skip → 즉시 종료, 메인 Claude 에 "산출물 보존됨, skip 처리" 보고
     - 재생성 → 버전 번호 +1 부여 후 정상 진행
   - 있음 + 사용자 명시 ("새로 만들어", "버전 업그레이드") → 버전 +1 부여 후 정상 진행

## 6. 예외·특별 처리

### 6-A. 정책 미적용 (매번 갱신·덮어쓰기 OK)

- 빌드 로그 `build_log.md`
- 임시 분석 파일 `analysis-tmp/*`
- 메모리·정책 파일 자체 수정
- WOL fetch 캐시 (`_wol_cache_*.md`, 15분 TTL)

### 6-B. 누적 보존 (덮어쓰지 말고 버전 누적)

검수 흔적은 4단/6단 방어 추적 자료라 덮어쓰면 안 된다. 재호출 시 누적 보존:

| 파일 | 누적 형식 |
|---|---|
| `_selfcheck.md` (서브 자체 검수) | `_selfcheck_v{N}.md` 누적 |
| `_selfcheck_script.md` (script 자체 검수) | `_selfcheck_script_v{N}.md` 누적 |
| `_planner_review_research.md` (Planner 1차) | `_planner_review_research_v{N}.md` 누적 |
| `_planner_review_script.md` (Planner 2차) | `_planner_review_script_v{N}.md` 누적 |

규칙: 첫 호출은 `_selfcheck.md`, 두 번째는 기존을 `_selfcheck_v1.md` 로 rename 후 새로 `_selfcheck.md` 생성. 가장 최신은 항상 버전 번호 없는 이름 유지.

### 6-C. 보조 리서처 산출 파일명 — 호출자 접두사 (필수)

같은 주차에 여러 planner 가 동시에 보조 리서처를 호출할 때 파일 이름 충돌을 막기 위해 모든 산출 파일명에 호출자 접두사를 부여:

- `treasures_isa-57-13.md` (treasures-talk-planner 가 호출)
- `gems_isa-57-13.md` (spiritual-gems-planner 가 호출)
- `publictalk_132_isa-57-21.md` (public-talk-builder 강연번호 132)

호출자 코드 표는 각 보조 리서처 정의 파일의 "산출 파일명 호출자 접두사" 단락 참조.

### 6-D. _v_old/ 폴더 누적 (추후 정리)

매주 "버전 업그레이드" 명시 시 기존 docx·pdf 가 `_v_old/` 로 이동. 매주 약 3MB/연 150MB. 즉시 위험은 없지만 장기적 정리 필요 — 30일 이상 된 `_v_old/` 자동 정리 hook 추후 도입 검토.

## 7. 핵심 메시지

**기본은 보존. 새로 만드는 건 사용자가 명시할 때만.**

이 정책의 목적은 두 가지:
1. 이미 검수·발송된 산출물이 후속 작업에 의해 의도치 않게 덮어쓰여 사고 나는 것 방지
2. 일괄 스킬(`/weekly` 등) 이 효율적으로 — **없는 것만** 채워 넣도록 유도
